35. A pod is stuck Terminating. Explain finalizers and the grace period, then force-delete it (--grace-period=0 --force) and state the risks

# Create a pod
kubectl run stuck-pod --image=docker.io/library/nginx:1.25

kubectl delete pod stuck-pod

# Force delete

# If finalizers blocking — remove them
kubectl patch pod stuck-pod -p '{"metadata":{"finalizers":null}}'

Explanation:
Grace period — when you delete a pod, Kubernetes sends SIGTERM and waits 30 seconds (default) for the app to shut down cleanly before sending SIGKILL. If the app doesn't respond it stays Terminating.
Finalizers — hooks that must complete before deletion is allowed (e.g. releasing volumes, deregistering from load balancer). If a finalizer hangs, pod stays Terminating forever.
Risks of force delete:
Data corruptionapp - didn't flush writes before killed
Duplicate pods - old and new pod running simultaneously
Volume not released - PVC may stay attached to deleted pod
Finalizers skipped - cleanup hooks never ran

kubectl rollout status deployment/stuck-deploy
# Waiting for deployment "stuck-deploy" to finish...
# error: progress deadline exceeded
# Diagnose
kubectl describe deployment stuck-deploy | grep -A10 Conditions
# Type: Progressing
# Reason: ProgressDeadlineExceeded
kubectl describe deployment stuck-deploy | grep -A10 Events
# pods failing to pull image
kubectl get pods -l app=stuck-deploy
# STATUS: ImagePullBackOff
# Remediate option 1 — rollback to previous
kubectl rollout undo deployment/stuck-deploy
# Remediate option 2 — fix the image
kubectl set image deployment/stuck-deploy nginx=docker.io/library/nginx:1.25
# Verify
kubectl rollout status deployment/stuck-deploy
kubectl get pods -l app=stuck-deploy


Explanation: ProgressDeadlineExceeded means the rollout didn't complete within the deadline (default 600 seconds). The old pods keep serving traffic while new pods fail — Kubernetes never cuts over until the new pods are Ready.

43. Compare OpenShift Route to Ingress.

Ingress is the standard Kubernetes way to expose HTTP/HTTPS services — it requires a separately installed ingress controller (nginx, traefik) to actually work. You define rules for routing traffic based on hostname or path.
OpenShift Route is OpenShift's built-in equivalent — it works out of the box with the HAProxy router that comes pre-installed. It supports the same hostname/path routing but also adds TLS termination, edge/passthrough/reencrypt modes, and automatic certificate management.

Summary: Route is more feature-rich and works out of the box on OpenShift, while Ingress is the portable standard that works on any Kubernetes cluster but needs extra setup.
