6. Set the update strategy to RollingUpdate with maxSurge: 1 and maxUnavailable: 0, and explain the effect on availability during an update.

rjesenje: editaj yaml file

Setting the Deployment strategy to RollingUpdate with maxSurge: 1 and maxUnavailable: 0 ensures zero downtime during updates. Kubernetes creates one additional pod with the new version before terminating an old pod. Since no pods are allowed to become unavailable (maxUnavailable: 0), the desired number of available replicas is maintained throughout the entire update process, providing continuous service availability.

7. Switch a Deployment's strategy to Recreate and describe a concrete scenario where this is required instead of RollingUpdate

dodaj u yaml: 
spec:
  strategy:
    type: Recreate

samo type: Recreate i obrisi cijeli rolling updates blok


Recreate terminates all existing Pods before creating new ones. Unlike RollingUpdate, there is a period of downtime because old and new versions never run simultaneously. A typical use case is an application that is incompatible with previous versions, such as when a database schema change requires all old instances to be stopped before the new version starts. This prevents conflicts and data corruption that could occur during a rolling update.

42. Inspect a Service's Endpoints/EndpointSlice and explain how they're populated from the pod selector.

kubectl create deployment webserver --image=docker.io/library/nginx:1.25 --replicas=3
kubectl expose deployment webserver --port=80 --target-port=80 --name=webserver-svc

# Check endpoints
kubectl get endpoints webserver-svc
kubectl describe endpoints webserver-svc

# Or check EndpointSlice
kubectl get endpointslice
kubectl describe endpointslice

How they're populated:
The kube-controller-manager watches for Pods that match the Service's selector — in this case app=webserver. Every time a Pod with that label becomes Ready, its IP is automatically added to the Endpoints list. When a Pod is deleted or becomes NotReady, its IP is removed. This is how traffic is routed — the Service is just a stable VIP, the Endpoints hold the actual Pod IPs behind it.


43. Demonstrate cross-namespace access using the FQDN, and show the short name fails from another namespace

# Create two namespaces
kubectl create namespace ns-a
kubectl create namespace ns-b

# Create deployment and service in ns-a
kubectl create deployment webserver --image=docker.io/library/nginx:1.25 -n ns-a
kubectl expose deployment webserver --port=80 --name=webserver-svc -n ns-a

# Test short name from ns-b - FAILS
kubectl run dns-test --image=docker.io/library/busybox:1.36 --rm -it --restart=Never -n ns-b -- nslookup webserver-svc

# Test FQDN from ns-b - WORKS
kubectl run dns-test --image=docker.io/library/busybox:1.36 --rm -it --restart=Never -n ns-b -- nslookup webserver-svc.ns-a.svc.cluster.local

Explanation:
Short name webserver-svc only resolves within the same namespace because DNS search appends the local namespace suffix automatically. From ns-b, the short name looks for webserver-svc.ns-b.svc.cluster.local which doesn't exist. The FQDN webserver-svc.ns-a.svc.cluster.local works from anywhere in the cluster because it specifies the exact namespace explicitly.

44. Compare kubectl port-forward to a Service versus to a Pod — explain the difference and when each fits.

kubectl create deployment webserver --image=docker.io/library/nginx:1.25
kubectl expose deployment webserver --port=80 --name=webserver-svc

# Port-forward to SERVICE
kubectl port-forward service/webserver-svc 8080:80 &
curl localhost:8080

# Port-forward to POD
kubectl port-forward pod/$(kubectl get pod -l app=webserver -o jsonpath='{.items[0].metadata.name}') 8081:80 &

# Stop both

When each fits:

Service — use when you just want to test the app locally and don't care which pod handles it
Pod — use when you need to debug a specific pod, check its logs, or reproduce an issue on that exact instance

53. Explain the default behavior when no probes are defined at all.

When no probes are defined, Kubernetes considers the container ready immediately after it starts and never restarts it unless the process crashes. This means a slow-starting app will receive traffic before it's ready, and a hung app that hasn't crashed will never be restarted.
